Why it matters

There's a better way than
connecting Power BI directly
to Epicor.

Most Epicor reporting starts the same way — Power BI pointed straight at the Epicor database or a BAQ, pulling data on demand. For a system representing a $500K–$2M+ investment, the out-of-box reporting experience is surprisingly painful. It works at first. Then it doesn't.

✕
Direct connection approach
Power BI → Epicor database or BAQ
—

Slow reports, angry users

Every refresh hits the live Epicor database. Complex visuals trigger complex queries. Reports time out. Users refresh repeatedly. The ERP slows down for everyone else.

—

Epicor schema complexity

The Epicor database has hundreds of tables with non-obvious relationships. DAX measures end up doing work that SQL should do — joins, case logic, cross-table calculations — making models fragile and hard to maintain.

—

No history, no comparisons

Epicor stores current state. Without a warehouse capturing snapshots, period-over-period analysis, budget vs. actual, and prior-year comparisons require complex workarounds — if they're possible at all.

—

Breaks on Epicor upgrades

Direct schema connections are brittle. When Epicor changes a table structure, columns move, or a BAQ is modified, every downstream report that touches it breaks — often silently.

—

One report at a time

Each report is its own island. No shared logic, no reusable measures, no consistent definitions. "Revenue" means something different in the Sales report than it does in the Finance report.

vs
✓
Tailored.BI Data Warehouse
Power BI → Fabric warehouse → Epicor
+

Fast reports, always

Power BI reads from pre-shaped, indexed warehouse tables — not live Epicor. Incremental refresh means only changed rows move. Reports open in seconds. The ERP is untouched.

+

Clean, BI-ready schema

A purpose-built star schema translates Epicor's complexity into simple, predictable tables. DAX measures are clean and fast. New reports can be built in hours, not days.

+

Full history, any comparison

The warehouse captures period snapshots and preserves history. Budget vs. actual, prior-year comparisons, rolling 12-month trends, and retail 4-4-5 calendar analysis all work natively.

+

Isolated from Epicor changes

The warehouse is the buffer. When Epicor is upgraded, only the data pipeline layer needs updating — not every report. Reports keep working while the plumbing is fixed behind the scenes.

+

One version of the truth

Every report shares the same dim tables, the same DAX measures, and the same definitions. Revenue is Revenue. Margin is Margin. Across Finance, Sales, Operations — everywhere.

Frequently asked questions

Do I need a Power BI license?

Yes — each person who views reports needs a Power BI Premium Per User (PPU) license at $20/user/month. PPU unlocks paginated reports for print-ready invoices and statements, higher refresh frequency, and advanced analytics — worth the extra $10/user over standard Pro. We include this in your all-in monthly cost estimate.

Who owns the data?

You do. Your data lives in an isolated Microsoft Fabric workspace. We are the administrators, but your data is contractually yours. We sign an NDA before any connection is made.

Does my IT team need to be involved?

Minimally. We need one read-only firewall rule opened to your Epicor server. That's typically a 15-minute task for your IT person. We provide the exact specification.

What if I want to cancel?

30 days written notice. We export your data, provide all report files, and help you transition. No hostage situations — if we're not delivering value, you should be able to leave.

How long does setup take?

Typically 3–4 weeks from signed agreement to first live report. Week 1: connection and warehouse setup. Week 2: pipeline validation. Week 3: report build and review. Week 4: training and go-live.

Do you work with Epicor on-premises?

Yes — that's our primary configuration. Epicor Kinetic on-premises with a read-only SQL connection through our cloud gateway. Cloud-hosted Epicor also works. We support both.

Architecture

Two data pipeline paths.
One data warehouse.

An interchangeable ingestion architecture means the warehouse schema never changes based on how data gets there. Switch paths mid-engagement and nothing downstream breaks.

Lane 1 — SQL Server path
1

Epicor Kinetic (production)

Live Epicor instance on the client network. Read-only access via linked server or direct SQL against the Epicor application database.

↓
2

TailoredBI local SQL Server

Stored procedures read from Epicor, apply watermark filtering on SysRevChangedDate, shape the data, and stage it for Fabric. Heavy transformation happens here — close to the source.

↓
3

On-premises data gateway

Installed on the always-on SQL Server machine. Bridges the local network to Fabric without opening inbound firewall ports.

↓
4

Fabric pipeline — SQL mode

Calls EXEC usp_Load_factGL @LastLoad, receives a clean shaped result set, MERGEs into the warehouse, updates the watermark.

or
Lane 2 — REST API path
1

Epicor Kinetic (cloud-hosted)

Internet-accessible Epicor instance. No local network access required — just valid API credentials and an API key scoped to the BAQs.

↓
2

Epicor BAQ REST endpoint

External BAQs exposed via /api/v2/odata/BaqSvc. Each accepts a SysRevChangedDate parameter so only changed rows are returned per refresh.

↓
3

No gateway needed

Epicor's cloud endpoint is internet-accessible. Fabric connects directly — no on-premises hardware or always-on machine required for this path.

↓
4

Fabric pipeline — API mode

REST Copy Activity with native @odata.nextLink pagination. Reads watermark, calls BAQ with date filter, auto-paginates, MERGEs rows, updates watermark.

TailoredBI Fabric Warehouse

Both lanes write to the same schema using the same MERGE pattern, controlled by the same staging.Watermark table. Power BI models, DAX measures, and TMDL calendars work identically regardless of which lane delivered the data.
